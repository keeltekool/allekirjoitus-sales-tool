# Identity Code Handling

The collection and handling of your signer's Personal Identity Code (SSN/HETU) is fully configurable - from disabling it entirely, to making it mandatory with strict identity matching, to cryptographically hashing it for GDPR data minimization. You control what gets collected, how it is stored, and who can override the setting.

- **Three configuration modes** - **Disabled**: the SSN field is completely hidden from the sender's interface, simplifying the UI for high-volume standard workflows. **Enabled (optional)**: the sender sees an optional SSN input field and checkbox per signatory, deciding case by case whether to enforce identity code collection. **Mandatory**: the sender is required to enter an SSN for each signatory before the system allows dispatch. When an SSN is provided, the platform enforces a strict identity match - if the SSN returned during the signer's bank authentication does not match the pre-entered value, the login fails and signing is blocked

- **Where the identity code appears** - by default, the signer's SSN is stored in plain text within the hidden XAdES XML metadata embedded inside the final PAdES document, but it does not appear on the visible cover page. The cover page can be customized to explicitly display the SSN in a designated column if your organization requires it (as used by Fennia Insurance, for example)

- **Cryptographic hashing** - instead of storing the plain-text SSN in the document, you can enable hashing so that only a cryptographic hash of the identity code is embedded in the signed PDF's XML. The actual SSN is stored separately in the authentication broker's internal logs for five years. This means the circulating document carries full legal validity (names, timestamps, IPs, and authentication methods are still recorded) while the sensitive identity code itself is protected. If a legal dispute requires absolute identity proof, the original SSN linked to the hash can be retrieved - but only via court order

- **Configuration levels** - SSN handling can be toggled per individual request by the sender in the portal. At the template or tenant level, administrators can lock the setting: for example, configuring the hashing checkbox to be visible and defaulted to on, but greyed out so senders cannot disable it. This enforces organization-wide privacy policy without relying on individual employee compliance
